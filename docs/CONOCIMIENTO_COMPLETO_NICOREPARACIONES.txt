========================================================================
1. IDENTIDAD DEL PROYECTO
========================================================================

Nombre del proyecto:
- NicoReparaciones

Tipo de sistema:
- Plataforma web integral para un negocio de reparacion de dispositivos
- Tienda online de productos y electronica
- Panel administrativo interno para operacion del negocio

Objetivo principal:
- Unificar en una sola plataforma la venta de productos, la gestion de
  reparaciones, la comunicacion automatica con clientes y la operacion
  interna del local.

Problema que resuelve:
- Evita tener sistemas separados para tienda, reparaciones y admin
- Centraliza configuracion del negocio, branding, ayuda, plantillas y
  trazabilidad
- Permite operar el negocio desde una sola interfaz web

Usuarios del sistema:
- Visitante publico
- Cliente autenticado
- Administrador

Capas funcionales:
- Publico: tienda, ayuda, consulta publica de reparacion
- Usuario autenticado: pedidos, reparaciones, cuenta
- Admin: operacion completa del negocio

========================================================================
5. POR QUE SE ELIGIO ESTE NUEVO STACK
========================================================================

Razones de arquitectura:

1. React
- Permite manejar pantallas complejas con alto nivel de interaccion
- Facilita reutilizar componentes
- Es mucho mas comodo para vistas admin grandes, formularios, tablas y
  estados visuales complejos

2. TypeScript
- Endurece contratos de datos
- Reduce errores por tipos
- Da consistencia entre frontend, backend y base de datos
- Es especialmente util en auth, pedidos, reparaciones, pricing y settings

3. NestJS
- Ordena el backend por modulos
- Facilita separar auth, store, admin, repairs, orders y pricing
- Mejora mantenibilidad y escalabilidad

4. Prisma
- ORM tipado
- Migraciones controladas
- Mejor relacion entre codigo y modelo de datos

5. PostgreSQL
- Mejor base relacional para reglas, catalogos y crecimiento operativo
- Muy adecuada para reporting, relaciones complejas y consistencia

6. Zod
- Validacion fuerte y declarativa
- Muy util para formularios, payloads de API y contratos compartidos

7. Tailwind CSS
- Se mantuvo por su velocidad para paridad visual y control fino de UI

8. Vite
- Desarrollo y build rapidos
- Buen encaje para React + TypeScript

========================================================================
6. ARQUITECTURA ACTUAL DEL SISTEMA
========================================================================

Arquitectura general:
- Frontend SPA en React
- Backend API REST en NestJS
- Persistencia con Prisma + PostgreSQL
- Assets publicos servidos por el frontend o por la API cuando aplica

Flujo general:
1. El usuario entra a una ruta del frontend
2. React resuelve la pantalla y renderiza la vista
3. La vista consume endpoints del backend bajo /api/*
4. NestJS recibe la request
5. El modulo correspondiente aplica auth, validacion y reglas
6. Prisma lee o escribe en PostgreSQL
7. El backend responde JSON tipado
8. El frontend actualiza estado y UI
9. Si corresponde, se dispara mail, branding o log operativo

Prefijo API:
- /api

URLs locales principales:
- Legacy: http://127.0.0.1:8000
- Next web dev: http://localhost:5174
- Next web preview/prod local: http://127.0.0.1:4174
- Next API: http://127.0.0.1:3001/api

Entrypoints de arranque local:
- nico-dev.bat next-setup
- nico-dev.bat next-start
- nico-dev.bat next-stop
- nico-dev.bat next-qa
- nico-dev.bat next-preprod
- nico-dev.bat next-close

22. CEREBRO DEL PROYECTO - RESUMEN CONCEPTUAL

Si alguien tiene que entender rapidamente como piensa este sistema:

1. El frontend React es la cara visible
- renderiza la experiencia
- navega
- protege rutas
- consume la API
- mantiene paridad visual

2. El backend NestJS es el cerebro operativo
- autentica usuarios
- valida permisos
- resuelve reglas de negocio
- coordina tienda, pedidos, reparaciones, catalogos y admin

3. Prisma + PostgreSQL son la memoria estructurada
- guardan usuarios, pedidos, reparaciones, productos, settings y logs

4. AppSetting y branding hacen que el sistema sea configurable
- no todo esta hardcodeado
- parte del comportamiento y del look sale de configuracion admin

5. QA y runbooks garantizan que el sistema sea operable
- no solo se migro codigo
- tambien se migro proceso, validacion y base de despliegue
